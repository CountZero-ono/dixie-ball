# Flatline ↔ Voice Bridge — Future Notes

## What exists today

Voice pipeline:  Ball → Wyoming → HA → Qwen3.6 (stateless, no memory)

The LLM answers but doesn't write to Flatline or read from it.
Every voice session starts fresh — Flatline's amnesia problem, applied to voice.

---

## What we want eventually

Voice queries should be able to:

1. **Read from Flatline memory** — "What did I decide about the ROCm issue?" pulls from L2 (MemMachine)
2. **Write observations** — "Note that the Piper voice sounds robotic, try Kokoro" → L1 observation
3. **Trigger Flatline flows** — "Sign me off" → calls `sign_off` MCP tool

---

## Approach

### Option A — HA custom intent → REST call to Flatline MCP server

`flatline_mcp_server.py` already runs on the SER7.
A thin FastAPI shim could expose a subset of MCP tools as HA REST commands:

```
POST http://192.168.1.112:8888/voice/write_observation
POST http://192.168.1.112:8888/voice/query_memory
```

HA automation: on specific voice intent → call REST → Flatline writes/reads.

### Option B — System prompt injection at conversation agent level

The HA conversation agent system prompt could include a dynamic context block
injected from Flatline's `flatline_briefing.md` (already generated by `hand_off`).

Read briefing at agent startup → prepend to system prompt → voice sessions
have relevant memory context without writing back.

### Option C — Wyoming conversation proxy

A custom Wyoming-compatible server that:
1. Receives STT text from HA
2. Queries Flatline L2 for relevant context
3. Constructs prompt with context + sends to Qwen3.6
4. Returns response + writes observation to L1

This is the cleanest architecture but most work.

---

## Decision (2026-07-02)

Not implementing now. Get the basic pipeline working first:
Ball → Whisper → Qwen → Piper. Then evaluate.

Option B (system prompt injection from briefing.md) is the lowest-effort first step
and already leverages existing Flatline infrastructure.
